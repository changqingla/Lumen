# 智能体工作流详解

> 本文档描述原子工具架构下，智能体处理不同任务类型的完整工作流。

---

## 一、总体架构

系统采用**原子工具 + ReAct 循环**架构。Agent 通过自主推理，组合调用原子级别的工具来完成各种任务。每个工具只做一件事，Agent 负责编排工具的调用顺序和组合方式。

核心理念：**原子工具，自主编排**。Agent 像一个研究员一样思考，根据用户问题的性质自主选择合适的工具组合，而非依赖预定义的复合流程。

### 可用原子工具

| 工具 | 用途 |
|------|------|
| `recall` | 语义检索——从用户的文档知识库中检索相关片段 |
| `web_search` | 联网搜索——通过 Tavily 搜索互联网信息 |
| `read_document` | 读取文档——按 doc_id 读取全文或按 section 关键词读取特定章节 |
| `read_document_outline` | 文档大纲——获取文档的标题结构和各章节字符数 |
| `write_note` | 写入笔记——向工作记忆（key-value 存储）写入内容 |
| `read_note` | 读取笔记——从工作记忆中读取内容（支持精确/模糊匹配、列表、全部读取） |
| MCP 工具 | 外部工具——通过 MCP 协议动态加载的外部工具（如 arXiv 论文搜索等） |
| `finish` | 结束推理——输出最终答案 |

所有工具都是原子级别的，每次调用只执行一个明确的操作。Agent 通过多轮 ReAct 循环自主组合这些工具来完成复杂任务。

---

## 二、ReAct 主循环

每次用户发送消息，系统执行以下流程：

### 1. 请求初始化

- 验证必要参数（API 密钥、模型配置、召回配置等）
- 创建运行时 LLM 实例和工具实例
- 加载或创建会话，获取历史对话上下文
- 通过 `set_document_context()` 注入文档元数据（文档名称和内容）

### 2. Thought → Action → Action Input → Observation 循环

Agent 进入迭代循环，每轮：

1. **Thought（思考）**：Agent 分析当前信息，决定下一步行动。思考过程实时流式输出给用户。
2. **Action（行动）**：Agent 选择一个原子工具。
3. **Action Input（输入）**：Agent 提供工具所需的输入参数。
4. **Observation（观察）**：工具执行后返回结果，Agent 将结果记录到 Scratchpad 中。

循环持续直到 Agent 调用 `finish` 工具提供最终答案。

### 3. 安全机制

循环受到多重保护：

- **最大迭代次数**：超过上限后强制结束，基于已收集信息生成答案
- **循环检测**：如果 Agent 连续多次调用相同工具和参数，判定为陷入循环，强制结束
- **连续错误检测**：工具调用连续失败超过阈值，强制结束
- **Token 预算**：Scratchpad 的 token 使用率超过阈值时，提示 Agent 尽快完成
- **无进展检测**：最近几次工具调用都未获取有效信息时，建议结束

### 4. 会话保存

循环结束后，用户消息和 Agent 回答都会保存到会话历史中，支持多轮对话。

## 三、不同任务类型的工作流

以下是 Agent 面对不同类型任务时的典型行为模式。Agent 通过自主推理选择工具组合，而非硬编码路由。

### 3.1 简单对话（无文档的简单问题）

**场景**：用户没有关联文档，提出一般性问题。

**典型流程**：
1. Agent 思考：用户的问题不需要文档检索，可以直接回答
2. Agent 调用 `finish`，直接给出答案

整个过程只需一轮迭代。

### 3.2 单文档问答

**场景**：用户关联了一篇文档，针对文档内容提问。

**典型流程**：
1. 调用 `read_document_outline` 获取文档结构和各章节大小
2. 根据大纲判断相关章节，调用 `read_document` 读取特定章节
3. 如果章节内容不够，调用 `recall` 进行语义检索补充信息
4. 信息充分后，调用 `finish` 生成答案

**提示**：
- 短文档（大纲显示总字符数 < 5000）可直接读取全文
- 长文档应先看大纲，再读取特定章节
- 当需要跨章节查找特定信息时，使用 `recall` 语义检索

### 3.3 单文档总结

**场景**：用户关联了一篇文档，要求总结。

**典型流程**：
1. 调用 `read_document_outline` 了解文档结构
2. 短文档：直接调用 `read_document` 读取全文
3. 长文档：逐章调用 `read_document` 读取各章节，每读完一章调用 `write_note` 记录要点
4. 调用 `read_note`（all）读取所有笔记
5. 调用 `finish` 生成结构化总结

**关键**：对于长文档，`write_note` 是核心工具——它让 Agent 能在有限的上下文窗口内处理超长文档，通过逐章阅读、记录要点、最后综合的方式完成总结。

### 3.4 多文档总结

**场景**：用户关联了多篇文档，要求总结。

**典型流程**：
1. 对每篇文档重复：`read_document_outline` → `read_document`（关键章节）→ `write_note`（记录要点）
2. 调用 `read_note`（all）读取所有文档的笔记
3. 调用 `finish` 生成综合总结报告

**提示**：
- 每篇文档重点关注：研究问题、方法论、主要发现、结论
- 最终总结应按主题组织，而非按文档逐篇罗列
- 注意提炼文档间的共性和差异

### 3.5 文献对比

**场景**：用户关联了多篇文档，要求对比分析。

**典型流程**：
1. 对每篇文档：`read_document_outline` → `read_document`（关键章节）
2. 对每篇文档调用 `write_note`，按统一维度记录（研究目标、方法、数据、结论、创新点）
3. 调用 `read_note`（all）读取所有笔记
4. 调用 `finish` 生成结构化对比分析（包含对比维度表格、异同点分析、各自优劣势）

### 3.6 综述生成

**场景**：用户关联了多篇文档，要求生成学术综述。

**典型流程**：
1. 对每篇文档：`read_document_outline` → `read_document`（关键章节）→ `write_note`（记录要点）
2. 调用 `read_note`（all）读取所有笔记
3. 调用 `finish` 生成学术综述（包含：引言、研究现状分类、各方向进展、研究空白、未来方向、结论）

**提示**：
- 按研究主题组织，而非按单篇论文罗列
- 引用文档名称作为参考文献
- 识别研究空白和未来方向

### 3.7 多文档问答

**场景**：用户关联了多篇文档，针对文档内容提出具体问题。

**典型流程**：
1. 调用 `recall` 跨所有文档进行语义检索
2. 如果召回结果不够，根据结果线索调用 `read_document` 读取相关文档的特定章节
3. 调用 `finish` 综合多篇文档信息生成回答，注明信息来源

**提示**：
- `recall` 适合快速定位跨文档的相关片段
- 如果需要更完整的上下文，再用 `read_document` 读取完整章节

### 3.8 联网搜索

**场景**：用户需要获取外部信息。

**典型流程**：
1. 调用 `web_search` 搜索互联网信息
2. 综合搜索结果，调用 `finish` 生成答案

Agent 也可以将联网搜索与文档工具结合使用，例如先检索文档内容再搜索外部信息补充，或反过来。

### 3.9 论文搜索（MCP 工具）

**场景**：用户要求搜索学术论文。

**典型流程**：
1. 调用 MCP 工具（如 `search_papers`），传入搜索关键词和时间范围
2. 获得论文列表后，格式化结果调用 `finish`

**时间敏感性规则**：
- 搜索"最新"论文时，`date_from` 设为当前日期前 6 个月
- 搜索"近期"论文时，`date_from` 设为当前日期前 12 个月
- 不使用硬编码日期

### 3.10 混合任务

Agent 可以在一次对话中自由组合任何工具。例如：
- `recall` + `web_search`：用外部知识补充文档信息
- `read_document_outline` + `read_document` + `recall`：先了解结构，再精读，再检索
- 多次 `read_document` + `write_note` + `finish`：系统性文档分析
- `web_search` + MCP 工具：综合多种外部信息源

这种灵活性是原子工具架构的核心优势——Agent 自主决策工具组合，根据实际情况动态调整策略。

## 四、SSE 事件流

整个处理过程通过 Server-Sent Events 实时推送给前端：

| 事件类型 | 说明 |
|---------|------|
| `thinking_start` / `thinking_end` | 思考过程的开始和结束标记 |
| `thought_chunk` | 思考过程的流式内容 |
| `answer_chunk` | 最终答案的流式内容 |
| `final_answer` | 最终答案（含 answer 和 session_id） |
| `token_usage` | Token 用量统计 |
| `error` | 错误信息 |
| `cancelled` | 用户取消生成 |

前端根据事件类型分别处理：思考过程显示在可折叠区域，最终答案显示在主内容区。

---

## 五、会话与上下文管理

- 每次对话使用 session_id 标识，支持多轮对话
- 会话历史存储在 PostgreSQL + Redis 中
- Agent 在每轮推理时都能看到历史对话上下文
- 文档元数据通过 `set_document_context()` 注入，Agent 在 prompt 中可以看到可用文档列表
