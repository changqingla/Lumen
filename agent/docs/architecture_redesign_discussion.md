# Agent 架构重设计讨论

## 背景

当前系统采用 Pipeline + ReAct 混合架构，存在以下问题：

1. **架构割裂**：根据意图类型硬编码路由，维护成本高
2. **Pipeline 僵化**：预定义线性流程，无法动态调整
3. **上下文管理困难**：多文档场景容易超限
4. **缺乏真正并行**：本质上还是串行思维

## 目标

设计一个更智能、更灵活的 Agent 架构。

## 讨论记录

### 2026-01-05 讨论

#### 提出的方案：ReAct 主循环 + 子 Agent

**核心思想**：
- 以 ReAct 为主循环，统一所有任务类型
- 引入子 Agent 机制处理复杂子任务
- 子 Agent 维护独立上下文，避免超限
- 支持异步并行执行

**示例场景 - 多文档总结**：
```
用户: 帮我总结这5篇论文

主 Agent (ReAct):
  Thought: 用户需要总结5篇论文，我应该并行处理每篇文档
  Action: spawn_sub_agents
  Action Input: {
    "task_type": "document_summary",
    "documents": ["doc1", "doc2", "doc3", "doc4", "doc5"],
    "parallel": true
  }
  
  [子 Agent 1-5 并行执行，各自维护独立上下文]
  
  Observation: 
    - doc1 总结: ...
    - doc2 总结: ...
    - ...
  
  Thought: 所有文档总结完成，现在需要综合这些总结
  Action: synthesize
  Action Input: {"summaries": [...]}
  
  Final Answer: 综合总结...
```

#### 待讨论的问题

1. **子 Agent 的定义**
   - 子 Agent 是否需要完整的 ReAct 能力？
   - 还是简化版的单任务执行器？

2. **通信机制**
   - 主 Agent 如何启动子 Agent？
   - 子 Agent 结果如何返回？
   - 是否需要中间状态同步？

3. **上下文管理**
   - 子 Agent 的上下文窗口大小如何配置？
   - 主 Agent 如何管理汇总后的信息？

4. **错误处理**
   - 子 Agent 失败时如何处理？
   - 是否需要重试机制？

5. **流式输出**
   - 如何向用户展示子 Agent 的进度？
   - 并行执行时如何合并流式输出？

### 待讨论的问题 (2026-01-05 继续)

#### 问题 1: 任务处理策略 - 硬编码 vs LLM 自主决策

**当前草案问题**：
草案中硬编码了"文献总结→创建子Agent"、"综述生成→先并行总结再综合"等规则，这违背了"让 LLM 更智能"的初衷。

**讨论点**：
- 是否应该完全由 LLM 决定如何处理任务？
- 如果是，我们只需要提供工具，让 LLM 自己判断何时使用
- 但这样会不会导致 LLM 做出低效的决策？（比如串行处理本可以并行的任务）

#### 问题 2: 阅读全文工具设计

**需求**：
- 封装一个从 MinIO 读取文档全文的工具
- 让 LLM 决定何时读取哪个文档

**挑战**：
- 几十篇文档时，如何让模型正确选择文档？
- 文档路径/ID 如何呈现给模型？
- 是否需要先提供文档列表（标题、摘要）让模型选择？

**可能的方案**：
1. 工具输入：`read_document(doc_id)` - 需要模型知道 doc_id
2. 先调用 `list_documents()` 获取文档列表，再选择性读取
3. 在 prompt 中直接注入文档列表（ID + 标题 + 简短描述）

#### 问题 4: Token 计数问题

**当前问题**：
项目中的 token 计数不准确，使用的是估算方法。

**建议方案**：
- 使用模型 API 返回的实际 token 数（`usage.prompt_tokens`, `usage.completion_tokens`）
- 在流式响应结束后获取实际使用量
- 更新 session 的 token 统计

**影响范围**：
- `agent_system/context/token_counter.py`
- `agent_system/context/session_manager.py`
- 所有 LLM 调用的地方
